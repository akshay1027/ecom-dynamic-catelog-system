     Plan: Fix Workflow Enforcement Gaps

     Context

     The user has an elaborate Claude workflow (TDD, pipeline, skills, hooks)
     but it breaks in new sessions. The example given: TDD was followed in
     session 1 but dropped in later sessions. The goal is to make the workflow
     self-enforcing — a new session with no prior context should still follow
     the same rules.

     ---
     Root Cause Audit

     Gap 1 — Pipeline direction CONFLICT (biggest issue)

     Global ~/.claude/CLAUDE.md says:
     implement → /write-tests → /review-code → fix P0/P1 → /create-pr
     This means: write code first, then write tests. This is not TDD.

     Project CLAUDE.md says:
     TDD Contract: Write failing test FIRST (RED). Confirm it fails. Write
     minimum code to pass (GREEN). Refactor.
     Two standing orders that directly contradict each other on the most
     fundamental step. In a new session, Claude has no history and both
     instructions are weighted equally — whichever reads last or feels more
     "natural" wins. The global pipeline wins often because it's longer and more
      detailed.

     Gap 2 — TDD is described, not commanded

     The TDD Contract in project CLAUDE.md reads like documentation:

     ▎ "PostToolUse hooks auto-run tests on every save. Stop hook blocks exit if
     ▎  tests are red."

     This describes what hooks do, not what Claude must do. Imperative language
     ("NEVER write implementation without a failing test first. This is
     blocking.") is much harder for the model to ignore than descriptive
     language.

     Gap 3 — Hooks enforce test-passing, not TDD sequence

     The PostToolUse hook runs npm test after every file edit — great for
     regressions.
     The Stop hook blocks exit if tests are red — great for not shipping broken
     code.
     But neither hook checks if tests were written BEFORE implementation. Hooks
     can enforce a state (tests pass), not a sequence (test first, then code).

     Gap 4 — No TDD in memory

     Project memory has:
     - feedback_github_mcp.md — use MCP tools
     - project_ecom_catalog.md — project overview

     No TDD memory entry. Memory is loaded at session start and actively
     consulted — adding TDD there would reinforce it alongside the CLAUDE.md
     instruction.

     Gap 5 — /ship-feature skill is optional, not auto-invoked

     The ship-feature.md skill encodes the correct end-to-end workflow including
      TDD, plan mode, review, PR. But it only runs when explicitly invoked. If
     the user says "add X" without /ship-feature, Claude falls back to its
     default reading of the global pipeline (implement first).

     Gap 6 — Global pipeline is authoritative but wrong for TDD

     The global CLAUDE.md says:

     ▎ "This pipeline always runs. If I only ask for implementation, you still 
     ▎ complete the full pipeline unless I explicitly say to stop after a step."

     The "full pipeline" starts with implement, which means: even when following
      the spirit of this rule, TDD is being violated. The authoritative
     instruction works against TDD.

     ---
     What Changes

     Fix 1 — Rewrite the pipeline in global CLAUDE.md to be TDD-first

     File: ~/.claude/CLAUDE.md

     Change:
     implement → /write-tests → /review-code → fix P0/P1 → /create-pr
     To:
     write-failing-test (RED) → implement (GREEN) → /write-tests (full coverage)
      → /review-code → fix P0/P1 → /create-pr

     And add an explicit standing order under Automated Workflow Rules:
     **TDD is non-negotiable on any code change:**
     Before writing any implementation code: (1) write a failing test for the
     specific behaviour, (2) run the suite and confirm it fails (RED), (3) write
      minimum code to make it pass (GREEN), (4) then run /write-tests for full
     coverage.
     NEVER write implementation code before a failing test exists. This is
     blocking — same as idempotency and no silent failures.

     Fix 2 — Strengthen TDD contract in project CLAUDE.md

     File:
     /Users/akshayrr/Documents/projects/ecom-dynamic-catelog-system/CLAUDE.md

     Change the TDD Contract from descriptive to imperative:
     ## TDD Contract
     NEVER write implementation code before a failing test exists for it.
     Sequence (no exceptions): RED (write failing test, confirm it fails) →
     GREEN (minimum code to pass) → REFACTOR.
     PostToolUse hooks auto-run tests on every save. Stop hook blocks exit if
     tests are red.

     Fix 3 — Add a PostToolUse reminder hook for implementation edits

     File: /Users/akshayrr/Documents/projects/ecom-dynamic-catelog-system/.claud
     e/settings.json

     Add a second PostToolUse hook entry that fires when Claude edits
     implementation files (non-test files in backend/src/ or frontend/src/) and

     FILE="$CLAUDE_TOOL_INPUT_FILE_PATH"
     if [[ "$FILE" == */backend/src/* || "$FILE" == */frontend/src/* ]]; then
       if [[ "$FILE" != *.test.js && "$FILE" != *.test.jsx && "$FILE" !=
     *.spec.* ]]; then
         echo "TDD CHECK: Did you write and confirm a failing test BEFORE this 
     edit? If not, stop and write it now."
       fi
     fi
     This doesn't block (no exit 2) but it fires on every implementation edit,
     forcing Claude to verify TDD order before proceeding. Blocking would cause
     false positives for legitimate refactors.
