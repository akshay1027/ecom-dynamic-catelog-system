     v5: Dynamic Variant Options

     Context

     v4 (size-only variants) is complete. Now brands should be able to define
     their own variant TYPES (colour, size, material, etc.) and specify multiple
      VALUES per type (e.g., colour → blue, green, yellow), each with individual
      stock. The backend is already fully generic — no changes needed there. All
      changes are in the admin form and the display components.

     User's mental model:
     - "Variant" = a variant type/dimension (colour, size, material)
     - "Options for each variant" = the values under that type (blue, green,
     yellow)
     - Each value is a separate backend variant: { options: { colour: 'blue' }, 
     stock: 10 }
     - Single-key variants only in this phase (multi-dimensional combinations
     like size+colour deferred)

     ---
     Backend — no changes

     inMemoryStore.js already handles any options key via isVariantDimension
     routing in search(). serializeSchema already exposes isVariantDimension.
     Variant CRUD endpoints already accept { options: { [key]: value }, stock }.

     Seed data update — 3 apparel products get colour as a second variant type
     (each colour value = separate variant, single-key options):
     - Slim Fit Graphic Tee: keep existing size variants + add colour variants {
      options: { colour: 'black' }, stock: 40 }, { options: { colour: 'white' },
      stock: 30 }. Remove color from attributes.
     - Performance Running Shorts: same pattern (black/navy colour variants).
     - Floral Midi Dress: same pattern (floral pink/floral blue colour
     variants).

     function buildVariantTypes(variants = []) {
       const typeMap = new Map();
       for (const v of variants) {
         for (const [key, val] of Object.entries(v.options || {})) {
           if (!typeMap.has(key)) typeMap.set(key, []);
           typeMap.get(key).push({ id: v.id, value: String(val), stock:
     String(v.stock) });
         }
       }
       return [...typeMap.entries()].map(([key, values]) => ({ key, values }));
     }
     // In buildInitialState:
     variantTypes: buildVariantTypes(product?.variants)

     New handlers:
     - addVariantType() → appends { key: '', values: [] } to variantTypes
     - updateVariantTypeKey(typeIdx, key) → updates variantTypes[typeIdx].key
     - removeVariantType(typeIdx) → removes that type
     - addVariantValue(typeIdx) → appends { id: null, value: '', stock: '0' } to
      variantTypes[typeIdx].values
     - updateVariantValue(typeIdx, valIdx, field, val) → updates value or stock
     - removeVariantValue(typeIdx, valIdx) → removes that value row

     Section title/button: "Variants" / "+ Add Variant Type"

     ---
     ADR

     ADR-009: Dynamic Variant Form UX — "Variant type with value list" model
     (chosen) vs "Per-variant row with key-value pairs". Tradeoff: the
     type+values model is more intuitive for brand admins (matches how
     Shopify/etc. present variants) and enforces consistent key names per type.
     The per-row approach is more flexible but leads to inconsistent option keys
      (one row has colour, another has color). Consequence: single-key variants
     only in storage (no multi-dimensional combinations in Phase 1).
