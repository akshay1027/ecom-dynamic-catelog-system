     Dynamic E-Commerce Catalog System — Implementation Plan

     Context

     Building a greenfield prototype of a dynamic e-commerce catalog that
     handles multiple product types (apparel, furniture, electronics) with
     per-type custom attributes. The goal is a clean, swappable architecture
     (in-memory now → DB later) with strict TDD workflow enforcement baked into
     the dev environment from day one.

     Directory: /Users/akshayrr/Documents/projects/ecom-dynamic-catelog-system/
     (currently empty)

     ---
     Architecture Decisions (ADRs to be written before code)

     Data Model (ADR-001)

     Flat attributes: {} JSON on every product — no typed sub-schemas, no EAV.
     Core fields are top-level; everything category-specific lives in
     attributes. Rejected: class hierarchy (bad for DB migration), nested
     attributes (complicates query parsing).

     API (ADR-002)

     RESTful at /api/v1/products. Attribute filtering via bracket notation:
     ?attributes[size]=M. Response envelope: { success, data, error } on all
     responses. No auth for prototype (document this decision in ADR-002).

     Frontend (ADR-003)

     React 19 + Vite 6. Co-located component tests (ProductCard.test.jsx next to
      ProductCard.jsx). No Next.js (SSR overkill), no CRA (deprecated).

     State Management (ADR-004)

     Local useState + custom useProducts(filters) hook. No Redux/Zustand/Context
      — one data domain doesn't need a global store. Hook interface is designed
     to be drop-in replaced with React Query when caching is needed.

     ---
     Product Schema (authoritative contract)

     {
       id: 'uuid-v4',          // generated, immutable
       name: 'string',          // required, 1-200 chars
       description: 'string',   // optional, max 2000 chars
       price: number,           // required, >= 0
       currency: 'USD',         // 3-char ISO 4217, default 'USD'
       category: 'string',      // required: 
     'apparel'|'furniture'|'electronics'|...
       type: 'string',          // required: 'shirt'|'sofa'|'laptop'|...
       images: ['url'],         // optional, default []
       stock: integer,          // required, >= 0
       tags: ['string'],        // optional, default []
       attributes: {},          // flat key-value: string|number|boolean values 
     only, no nesting
       createdAt: 'ISO-8601',   // set on create, never updated
       updatedAt: 'ISO-8601'    // set on create + every update
     }

     ---
     API Contract (ADR-002 summary)

     Method: POST
     Path: /api/v1/products
     Status: 201
     ────────────────────────────────────────
     Method: GET
     Path: /api/v1/products/:id
     Status: 200 / 404
     ────────────────────────────────────────
     Method: PUT
     Path: /api/v1/products/:id
     Status: 200 / 404 / 400
     ────────────────────────────────────────
     Method: DELETE
     Path: /api/v1/products/:id
     Status: 200 / 404
     ────────────────────────────────────────
     Method: GET
     Path: /api/v1/products?name=&category=&type=&minPrice=&maxPrice=&attributes
     [key]=val&tags=&page=&limit=
     Status: 200

     ---
     Store Interface (swappability contract)

     backend/src/store/inMemoryStore.js exports exactly:
     { create, findById, update, remove, search, clear }
     clear() is test-only. No file outside store/ touches the underlying Map.
     Swapping to Postgres = create postgresStore.js with same 5 exports, change
     one require() in routes.
