⏺ This is the right question to ask before building. Let me think through all
  the options properly.

  The core problem: a products table where each product has arbitrary key-value
  attributes, and we need to:
  1. Filter products by attribute values (exact, range, OR)
  2. Discover what attributes exist per category (for UI)
  3. Scale to millions of products

  Here are all the serious approaches against a Postgres backend:

  ---
  Approach 1: Plain JSONB Column

  CREATE TABLE products (
    attributes JSONB DEFAULT '{}'
  );

  Filtering:
  -- Exact (GIN containment — fast with index)
  WHERE attributes @> '{"color": "red"}'

  -- Range (requires expression index or it's a full scan)
  WHERE (attributes->>'ram_gb')::int BETWEEN 8 AND 32

  -- Multi-value OR
  WHERE attributes->>'color' = ANY(ARRAY['red','blue'])

  Schema discovery (for UI):
  SELECT DISTINCT jsonb_object_keys(attributes) FROM products WHERE category =
  'electronics';
  This is O(n) — full scan every time the user changes category.

  │     │                                                                    │
  ├─────┼────────────────────────────────────────────────────────────────────┤
  │ ✅  │ Simple, single column, zero schema changes for new product types   │
  ├─────┼────────────────────────────────────────────────────────────────────┤
  │ ✅  │ GIN index makes containment exact-match fast                       │

  ---
  Approach 3: JSONB + Attribute Registry (Recommended)

  Use JSONB for storage, but maintain a separate summary table that tracks what
  attributes exist, their types, and their value ranges. Updated on every
  product write.

  -- Main storage — same as Approach 1
  CREATE TABLE products (
    attributes JSONB DEFAULT '{}'
  );
  CREATE INDEX ON products USING GIN(attributes jsonb_path_ops);

  -- Registry: maintained as a side-effect of every product create/update/delete
  CREATE TABLE attribute_registry (
    category TEXT NOT NULL,
    attribute_key TEXT NOT NULL,
    attribute_type TEXT NOT NULL,       -- 'string' | 'number' | 'boolean'
    string_values TEXT[],               -- distinct values (capped at ~100)
    number_min DECIMAL,
    number_max DECIMAL,
    product_count INTEGER DEFAULT 0,
    PRIMARY KEY (category, attribute_key)
  );

  Schema discovery (O(1) — reads from registry, not products):
  SELECT * FROM attribute_registry WHERE category = 'electronics';
  -- Returns instantly regardless of product count

  Filtering (JSONB + GIN):
  -- Exact/boolean (GIN containment — O(log n))
  WHERE attributes @> '{"color": "red"}' AND category = 'apparel'

  -- Multi-value OR (GIN — O(log n) per OR branch)
  WHERE category = 'apparel' AND (
    attributes @> '{"color": "red"}' OR attributes @> '{"color": "blue"}'
  )

  -- Range (add expression index per numeric attr as needed)
  WHERE category = 'electronics' AND (attributes->>'ram_gb')::int >= 16
  -- With: CREATE INDEX ON products (((attributes->>'ram_gb')::int)) WHERE 
  category = 'electronics'

  Approach 4: Dedicated Search Engine (Elasticsearch / Typesense / Meilisearch)

  Sync products to a search service. All attribute filtering, faceting (schema
  discovery), and text search happens there. Postgres is source of truth.

  ├─────┼────────────────────────────────────────────────────────────────────┤
  │ ❌  │ Overkill for < 1M products                                         │
  ├─────┼────────────────────────────────────────────────────────────────────┤
  │ ❌  │ Wrong layer for a catalog API prototype — complexity without clear │
  │     │  benefit yet                                                       │
  └─────┴────────────────────────────────────────────────────────────────────┘

  ---
  Decision

  Approach 3 is the right architecture. Here's why it fits this project
  specifically:

  1. The in-memory prototype already implements the registry pattern — byBrand,
  byCategory, byType are hand-rolled secondary indexes. We just extend that to
  attributes: add attributeRegistry = Map<category, Map<key, {type, values, min,
   max}>>, updated on every create/update/remove. The shape of the in-memory 
  code becomes the spec for the Postgres implementation.
  2. The Postgres migration is straightforward: attributes JSONB + GIN index for
   the product table, attribute_registry table with upserts on write. The API
  contract (/api/v1/products/attributes/schema) stays identical.
  3. Range query performance is managed incrementally: Add CREATE INDEX ON 
  products (((attributes->>'ram_gb')::int)) WHERE category = 'electronics' only
  when query patterns demand it. Don't pay for indexes you don't need yet.
  4. The schema discovery endpoint becomes a catalog API that maps to a real
  product information management (PIM) pattern used in production e-commerce
  systems.

     Subagent 2 — Scalable Attribute Filters

     Goal: Dynamic per-category attribute schema endpoint + enhanced search
     (multi-value OR, numeric range) + frontend attribute filter UI with
     checkboxes and range inputs. Runs AFTER Subagent 1.

     Architectural approach: JSONB + Attribute Registry pattern

     The attribute filter system uses a registry-based design, not a scan-based
     one. This is the correct architecture for Postgres and mirrors the pattern
     cleanly in-memory.

     Why not scan products on every schema request:
     Scanning all products to derive the attribute schema on every
     /attributes/schema request is O(n). At 1M products, this is a 1-second
     query every time a user changes the category dropdown. Unacceptable.

     The registry pattern:
     - Maintain a secondary data structure (attributeRegistry) that is updated
     on every create/update/remove
     - Schema discovery reads from the registry in O(1)
     - This maps to a Postgres attribute_registry table (upserted on product
     writes) when we swap the store

     In-memory implementation (backend/src/store/inMemoryStore.js):
     // Module-scope registry: category → { [attributeKey]: { type, values: Set,
      min, max } }
     const attributeRegistry = new Map(); // 'electronics' → { color: { type: 
     'string', values: Set }, ram_gb: { type: 'number', min: 8, max: 32 } }

     function updateRegistry(category, attributes, oldAttributes = null) {
       if (!attributeRegistry.has(category)) attributeRegistry.set(category,
     {});
       const catRegistry = attributeRegistry.get(category);

       for (const [key, val] of Object.entries(attributes || {})) {
         if (!catRegistry[key]) catRegistry[key] = { type: null, values: new
     Set(), min: Infinity, max: -Infinity };
         const entry = catRegistry[key];

         if (typeof val === 'boolean') {
           entry.type = entry.type === null ? 'boolean' : entry.type;
         } else if (typeof val === 'number') {
           entry.type = entry.type === 'string' ? 'string' : 'number';
           entry.min = Math.min(entry.min, val);
           entry.max = Math.max(entry.max, val);
         } else {
           entry.type = 'string';
           entry.values.add(String(val));
         }
       }
     }

     // Called in clear() to reset registry
     function clearRegistry() { attributeRegistry.clear(); }

     In create(): call updateRegistry(product.category, product.attributes)
     after adding to store.
     In update(): call updateRegistry(updated.category, updated.attributes)
     after patch.
     In clear(): call clearRegistry().

     New exported function (extend the store's public interface):
     function getAttributeSchema(category) {
       if (!category) {
         // Merge all categories — for a "no category selected" view
         const merged = {};
         for (const [, catSchema] of attributeRegistry) {
           for (const [key, entry] of Object.entries(catSchema)) {
             if (!merged[key]) merged[key] = { ...entry, values: new
     Set(entry.values) };
             else {
               // Merge — type wins toward 'string' if mixed
               if (entry.type === 'string') merged[key].type = 'string';
               entry.values.forEach(v => merged[key].values.add(v));
               if (typeof entry.min === 'number') merged[key].min =
     Math.min(merged[key].min, entry.min);
               if (typeof entry.max === 'number') merged[key].max =
     Math.max(merged[key].max, entry.max);
             }
           }
         }
         return serializeSchema(merged);
       }
       return serializeSchema(attributeRegistry.get(category) || {});
     }

     function serializeSchema(schema) {
       const result = {};
       for (const [key, entry] of Object.entries(schema)) {
         if (entry.type === 'boolean') {
           result[key] = { type: 'boolean' };
         } else if (entry.type === 'number') {
           result[key] = { type: 'number', min: entry.min === Infinity ? 0 :
     entry.min, max: entry.max === -Infinity ? 0 : entry.max };
         } else {
           result[key] = { type: 'string', values: [...entry.values].sort() };
         }
       }
       return result;
     }

     Frontend B: useAttributeSchema hook

     File: frontend/src/hooks/useProducts.js — add alongside existing useBrands:
     export function useAttributeSchema({ category, type } = {}) {
       const [schema, setSchema] = useState({});
       const [loading, setLoading] = useState(false);
       const key = `${category || ''}|${type || ''}`;

       useEffect(() => {
         if (!category) { setSchema({}); return; }
         let cancelled = false;
         setLoading(true);
         productsApi.getAttributeSchema({ category, type })
           .then(body => { if (!cancelled) setSchema(body.data || {}); })
           .catch(() => { if (!cancelled) setSchema({}); })
           .finally(() => { if (!cancelled) setLoading(false); });
         return () => { cancelled = true; };
       }, [key]);

       return { schema, loading };
     }
